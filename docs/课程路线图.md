# WebQuest 课程路线图

## 1. 文档目标

本文档用于定义 `WebQuest` 的教学主线、知识地图、章节拆分、能力递进关系和首版实施策略。

它解决的不是“题目数量不够”的问题，而是“如何让用户在闯关过程中真正学到 Web 开发知识”的问题。

本文档将作为后续以下工作的统一依据：

- 关卡题库设计
- 判题规则设计
- 技能树节点设计
- 地图区域内容规划
- 数据库关卡种子设计

---

## 2. 教学定位

`WebQuest` 不是传统刷题平台，也不是纯剧情游戏，而是一套以 RPG 闯关机制包装的 Web 开发学习路径。

用户在产品中的成长目标不是“通过几道题”，而是逐步获得以下真实能力：

- 理解网页的结构组成
- 掌握页面样式和布局方法
- 能用 JavaScript 编写基础逻辑
- 能使用 DOM 完成交互开发
- 能独立完成小型前端页面和组件

一句话定义：

`WebQuest = 用游戏化闯关方式重构前端入门学习路径`

---

## 3. 课程设计原则

### 3.1 循序渐进

- 知识点必须按照依赖关系推进，不能跳跃。
- 用户必须先理解“页面结构”，再学习“页面表现”，最后学习“页面行为”。
- 每一关只聚焦 `1-2` 个核心知识点，避免信息过载。

### 3.2 重复巩固

- 每个关键知识点至少在 3 个不同场景中出现：
- 第一次：概念教学
- 第二次：模仿练习
- 第三次：独立应用

### 3.3 先局部，后综合

- 前几关只做小目标，例如“加一个标题”“设置一个背景色”。
- 中期关卡开始做组件，例如导航栏、卡片、表单。
- Boss 关再整合成一个完整页面或交互模块。

### 3.4 反馈必须可学习

- 失败反馈不能只返回“未通过”。
- 系统必须指出：
- 缺失了什么结构
- 少了什么样式
- 哪个交互没有实现
- 建议回顾哪一类知识

### 3.5 真实开发导向

- 题目内容要尽量贴近真实前端页面，而不是抽象语法题。
- 每章都要有“看得见成果”的关卡，增强成就感。

---

## 4. 总体学习主线

WebQuest 首版课程建议聚焦前端入门主线，按以下顺序推进：

1. `HTML 基础`
2. `CSS 基础`
3. `CSS 布局`
4. `JavaScript 基础`
5. `DOM 与事件`
6. `综合实战`

后续扩展路线：

7. `前端框架（React / Vue）`
8. `后端开发（Node / Express）`
9. `数据库（MySQL / Redis）`
10. `全栈项目实战`

---

## 5. 能力依赖图

### 5.1 核心依赖关系

```text
HTML 基础
   ↓
CSS 基础
   ↓
CSS 布局
   ↓
JavaScript 基础
   ↓
DOM 与事件
   ↓
综合实战
```

### 5.2 依赖解释

- `HTML 基础` 是一切页面开发的前提。
- `CSS 基础` 依赖 HTML 结构，负责让页面“可读、可看”。
- `CSS 布局` 依赖基础样式能力，负责让页面“能排版”。
- `JavaScript 基础` 负责让用户理解程序逻辑。
- `DOM 与事件` 是将 JavaScript 和页面连接起来的关键桥梁。
- `综合实战` 则是把结构、样式、逻辑和交互整合成真实页面。

## 6. 地图章节规划

建议将课程地图划分为 6 个教学区域。

### 6.1 新手村：HTML 基础

- 学习目标：认识网页结构与基础标签
- 关键词：`html`、文本、图片、链接、列表、表单、语义化
- 学习产出：能独立写一个基础静态页面

### 6.2 CSS 森林：CSS 基础样式

- 学习目标：掌握文字、颜色、背景、边框、盒模型
- 关键词：选择器、颜色、字体、背景、边距、圆角
- 学习产出：能把原始 HTML 页面美化成卡片式界面

### 6.3 布局峡谷：页面布局

- 学习目标：掌握常见页面布局方案
- 关键词：`display`、`flex`、`grid`、两栏布局、导航布局
- 学习产出：能完成常见页面骨架搭建

### 6.4 JS 风暴：JavaScript 基础

- 学习目标：掌握变量、条件、函数、数组、对象
- 关键词：逻辑判断、数据存储、函数封装
- 学习产出：能写基础程序逻辑

### 6.5 交互遗迹：DOM 与事件

- 学习目标：让网页具备交互能力
- 关键词：获取元素、修改内容、监听事件、读取输入
- 学习产出：能做按钮点击、计数器、表单反馈等小交互

### 6.6 前端试炼塔：综合实战

- 学习目标：将前面的知识整合成真实产品页面
- 关键词：组件化思维、页面拆分、结构 + 样式 + 交互
- 学习产出：能独立完成小型产品原型页面

## 7. 首版关卡规模建议

### 7.1 推荐规模

- 当前 `spec` 的 MVP 最低要求是：`至少 10 个前端学习关卡`
- 最小可用版：`24 关`
- 推荐教学版：`30 关`
- 更完整版本：`36 关`

### 7.2 当前建议

当前项目建议先按 `30 关` 进行课程规划，再分阶段落地。

这里需要区分两个层面：

- `产品 MVP 下限`：至少交付 10 关，满足演示与基础闭环
- `教学设计目标`：按 30 关规划，保证知识体系完整

也就是说，当前开发可以先上线前 10-12 关，但整体课程设计必须预先按更完整的结构搭好，避免后续返工。

章节分布建议如下：

| 章节 | 区域 | 关卡数 | 目标 |
|------|------|--------|------|
| 第一章 | 新手村 | 6 | 建立 HTML 基础认知 |
| 第二章 | CSS 森林 | 6 | 掌握样式基础 |
| 第三章 | 布局峡谷 | 6 | 掌握页面布局 |
| 第四章 | JS 风暴 | 5 | 掌握逻辑基础 |
| 第五章 | 交互遗迹 | 4 | 掌握 DOM 与事件 |
| 第六章 | 前端试炼塔 | 3 | 完成综合实战 |

总计：`30 关`

## 8. 关卡类型设计

为了让用户真正学会知识点，而不是依赖运气通关，建议每章采用多种题型混合。

### 8.1 教学关

- 作用：第一次引入知识点
- 特点：提示完整、目标清晰、限制少
- 适合场景：新概念首次出现

### 8.2 模仿关

- 作用：帮助用户照着示例完成
- 特点：有明确参考效果
- 适合场景：初步巩固

### 8.3 练习关

- 作用：让用户独立完成一个小任务
- 特点：提示减少，开始考查迁移能力
- 适合场景：中期巩固

### 8.4 修复关

- 作用：给出错误代码，让用户修复
- 特点：训练排错意识
- 适合场景：形成调试能力

### 8.5 综合关

- 作用：多个知识点结合完成一个模块
- 特点：接近真实开发
- 适合场景：章节收尾

### 8.6 Boss 关

- 作用：整合本章核心能力
- 特点：成果感最强
- 适合场景：章节终点

---

## 9. 章节内节奏模型

每一章建议遵循以下节奏：

1. `认知关`
2. `最小实践关`
3. `模仿关`
4. `独立练习关`
5. `半综合关`
6. `Boss 关`

这样可以保证：

- 用户先知道“是什么”
- 再知道“怎么做”
- 最后知道“什么时候该这样做”

## 13. 判题设计原则

为了保证平台具备真正的教学价值，后续判题系统不应仅使用 `requiredKeywords`。

建议逐步升级为以下 4 层：

### 13.1 结构判题

- 是否存在目标标签
- 是否存在关键属性
- 是否满足层级关系

### 13.2 样式判题

- 是否设置关键样式
- 是否使用正确布局方式
- 是否满足最小视觉要求

### 13.3 交互判题

- 是否绑定事件
- 是否修改 DOM
- 是否达成指定行为

### 13.4 引导反馈

- 缺失结构提示
- 样式缺失提示
- 交互未实现提示
- 推荐回顾章节

## 14. 与地图和技能树的映射策略

### 14.1 地图映射

- 每个区域对应一类知识主题
- 每个区域内部按 `sortOrder` 线性推进
- 章节 Boss 关通关后解锁下一区域

### 14.2 技能树映射

关卡与技能节点建议一一对应或多对一对应。

例如：

- `HTML 标签结构` ← 第 1、2 关
- `图片与链接` ← 第 3 关
- `CSS 颜色与边框` ← 第 8、9 关
- `Flex 布局` ← 第 13、14 关
- `事件监听` ← 第 26、27 关

这样做的价值：

- 技能树不是装饰页面，而是真实反映学习进度
- 技能节点可以反向提示用户还缺什么能力

## 16. 当前结论

WebQuest 的核心不是“做几道题”，而是“构建一条能让初学者真正掌握前端基础的游戏化学习路径”。

因此当前最合理的开发优先级是：

1. 先确定知识地图
2. 再设计详细关卡
3. 再重构关卡数据结构和判题逻辑
4. 最后再回到 UI 展示和高级系统扩展
