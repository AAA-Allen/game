# WebQuest Spec

## 1. 文档说明

- 项目名称：`WebQuest`
- 文档类型：统一规格说明文档 `Spec`
- 当前版本：`V1.0`
- 适用阶段：MVP 设计、前后端开发、联调与答辩展示
- 文档目标：整合产品需求、项目规范与后端架构，作为当前阶段唯一优先参考文档

## 2. 项目概述

`WebQuest` 是一个将 Web 开发学习路径与 RPG 游戏机制结合的沉浸式学习平台。用户通过世界地图探索、关卡闯关、在线编码、XP 成长与技能树解锁，在游戏冒险中学习 HTML、CSS、JavaScript 及后续全栈开发能力。

一句话定义：

`WebQuest = 用 RPG 游戏方式重构 Web 全栈开发学习体验`

## 3. 项目背景

传统 Web 开发学习模式通常依赖教程、视频与零散练习，存在以下问题：

- 学习路径枯燥，缺乏持续动力
- 知识点分散，难以形成完整成长路线
- 缺少即时反馈，成就感弱
- 初学者难以从语法学习过渡到实战开发

`WebQuest` 通过“地图探索 + 任务闯关 + 在线写代码 + 成长奖励”的闭环，提升用户的沉浸感、参与度与学习持续性。

## 4. 产品目标

### 4.1 总体目标

- 构建一个游戏化 Web 开发学习平台
- 帮助初学者系统掌握前端基础能力
- 提供可体验、可演示、可持续扩展的全栈学习平台原型

### 4.2 MVP 目标

- 实现世界地图基础界面
- 提供 10 个前端闯关关卡
- 支持 HTML/CSS/JavaScript 在线编辑与预览
- 实现用户登录、进度记录、XP 与等级系统
- 提供基础的后端 API 与提交流程

## 5. 目标用户

- Web 开发初学者
- 前端/后端入门学生
- 需要课程项目、毕业设计或答辩项目的开发者
- 希望通过互动玩法学习编程的兴趣用户

## 6. 产品定位与核心价值

### 6.1 产品定位

`WebQuest` 不是传统课程平台，而是以“闯关成长”为核心的 Web 技术学习产品。

### 6.2 核心价值

- 将抽象的技术学习路径可视化
- 将练习任务转化为关卡与奖励系统
- 通过即时反馈增强学习成就感
- 为后续扩展前端框架、后端、数据库与 AI 功能预留空间

## 7. MVP 范围

### 7.1 本阶段必须实现

- 世界地图 UI 基础版
- 新手村、CSS 森林、JS 风暴三个核心区域
- 10 个前端学习关卡
- 关卡详情页
- HTML/CSS/JS 在线代码编辑器
- 代码运行与实时预览
- 用户注册、登录、个人资料获取
- 提交记录、XP 计算、等级成长
- 关卡与区域解锁逻辑

### 7.2 本阶段不实现

- AI 教练系统
- 排行榜系统
- 多角色系统
- 复杂社交系统
- 真实后端代码隔离运行
- 高级粒子动画与复杂特效

## 8. 核心业务闭环

用户在平台内的核心使用流程如下：

1. 用户登录平台
2. 进入世界地图页
3. 选择已解锁区域与关卡
4. 查看任务背景、通关目标与提示
5. 在编辑器中编写代码
6. 运行并查看实时预览
7. 提交代码
8. 系统判定结果
9. 用户获得 XP、等级变化与关卡解锁反馈

## 9. 功能模块规格

### 9.1 用户系统

#### 目标

提供用户身份识别、登录态管理和学习进度归档能力。

#### 功能点

- 用户注册
- 用户登录
- 获取用户资料
- 保存等级、XP、通关关卡和解锁状态

#### MVP 约束

- 可使用简化注册登录流程
- 暂不支持第三方登录

### 9.2 世界地图系统

#### 目标

将 Web 技术学习路径按区域映射到游戏地图中，作为平台主导航入口。

#### 地图区域

- 新手村：HTML 基础
- CSS 森林：样式与布局
- JS 风暴：JavaScript 核心
- 前端框架城：预留
- 后端王国：预留
- 数据库遗迹：预留
- 全栈 Boss 战：预留

#### 功能点

- 地图展示
- 区域差异化视觉表现
- 解锁与未解锁状态展示
- 当前可挑战关卡高亮
- 点击进入关卡

### 9.3 关卡战斗系统

#### 目标

将每个 Web 开发任务设计为一个可完成、可评分、可奖励的游戏关卡。

#### 功能点

- 展示任务剧情与目标
- 展示通关提示
- 提供初始代码模板
- 支持运行和提交
- 返回通关结果、得分与奖励

#### 首批关卡建议

1. 创建基础 HTML 页面
2. 制作个人介绍卡片
3. 实现登录页面布局
4. 制作导航栏
5. 完成 Flex 布局页面
6. 实现按钮点击提示
7. 制作点击计数器
8. 实现表单输入读取
9. 完成 DOM 内容修改
10. 综合练习：交互式任务卡片

### 9.4 在线代码运行系统

#### 目标

让用户在浏览器中直接完成前端代码编写与预览。

#### 功能点

- HTML/CSS/JS 代码编辑
- 运行按钮
- 重置代码
- 预览窗口
- 基础错误提示

#### MVP 实现方式

- 采用前端 `iframe` 或沙箱机制实现 HTML/CSS/JS 预览
- 暂不支持真实 Java、Spring Boot、Node 服务端执行环境

### 9.5 XP 与等级系统

#### 目标

通过成长反馈驱动用户持续闯关。

#### 功能点

- 通关奖励 XP
- 失败不发放 XP
- 基础等级计算
- 关卡与区域解锁
- 通关结算反馈

### 9.6 技能树系统

#### 目标

可视化呈现用户的 Web 技术成长路径。

#### MVP 策略

- 首版先做展示型技能树
- 技能节点与关卡通关状态关联
- 暂不实现复杂属性系统

## 10. 页面规格

### 10.1 世界地图页

#### 页面目标

作为平台主入口，承载区域导航、进度信息与关卡入口。

#### 页面内容

- 地图背景
- 技术区域
- 解锁状态
- 用户等级与 XP
- 当前任务提示

### 10.2 关卡页面

#### 页面目标

承载剧情阅读、代码编写、运行预览与任务提交。

#### 布局建议

```text
┌──────────────┬──────────────┬──────────────┐
│ 任务剧情区   │ 代码编辑器   │ 实时预览区   │
└──────────────┴──────────────┴──────────────┘
```

#### 页面内容

- 关卡标题
- 背景剧情
- 通关目标
- 代码编辑器
- 运行按钮
- 提交按钮
- 实时预览区
- 成功/失败反馈

### 10.3 通关结算页

#### 页面目标

强化用户完成关卡后的奖励体验。

#### 页面内容

- 通关提示
- 星级评分
- XP 奖励
- 等级变化
- 解锁内容
- 下一关入口

## 11. 非功能需求

### 11.1 性能

- 页面主要交互响应流畅
- 预览反馈及时
- 首屏加载可接受

### 11.2 可用性

- 新手能快速理解页面功能分区
- 每关任务目标清晰
- 提交失败时必须提供明确反馈

### 11.3 安全性

- 用户数据必须绑定账号
- 奖励与 XP 逻辑由后端计算
- 所有受保护接口需进行鉴权

### 11.4 可扩展性

- 支持未来新增地图区域
- 支持扩展数据库、技能树、成就系统
- 支持后续接入真实后端代码运行环境

## 12. 技术架构规格

### 12.1 总体架构

项目采用前后端分离架构：

- 前端：`React + Vite + TypeScript`
- 后端：`Node.js + Express + TypeScript`
- 数据库：`MySQL` 或 `PostgreSQL`
- 鉴权方式：`JWT`

## 13. 后端架构规格

### 13.1 当前后端目标

MVP 后端负责：

- 提供统一 REST API
- 管理用户、区域、关卡、进度、提交
- 在后端进行通关结果和 XP 计算
- 为前端地图页、关卡页与个人信息页提供数据

### 13.2 当前后端实现策略

- 使用 `Express` 构建 API
- 使用 `TypeScript` 进行类型约束
- 使用 `mock-db.ts` 维护内存假数据
- service 层处理业务逻辑
- router 层处理接口映射
- middleware 处理鉴权、404 与错误响应

### 13.3 后端模块划分

- `auth`：注册、登录、JWT 返回
- `users`：用户资料
- `zones`：地图区域
- `levels`：关卡列表与详情
- `progress`：当前用户进度
- `submissions`：代码提交、规则校验、XP 计算

### 13.4 当前 API 规划

- `GET /health`
- `POST /api/auth/register`
- `POST /api/auth/login`
- `GET /api/users/profile`
- `GET /api/zones`
- `GET /api/levels`
- `GET /api/levels/:id`
- `GET /api/progress/current`
- `POST /api/submissions`

### 13.5 统一返回结构

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

错误格式：

```json
{
  "code": 404,
  "message": "Level not found",
  "data": null
}
```

## 14. 数据规格

### 14.1 核心数据对象

- 用户 `users`
- 地图区域 `zones`
- 关卡 `levels`
- 提交记录 `submissions`
- 用户进度 `user_progress`
- XP 日志 `xp_logs`
- 技能树节点 `skills`
- 用户技能记录 `user_skills`

### 14.2 字段规范

- 每张表必须具备主键 `id`
- 必须包含 `created_at` 和 `updated_at`
- 状态字段统一命名为 `status`
- 布尔字段使用 `is_` 前缀

### 14.3 约束规则

- 关卡必须归属某个区域
- 提交记录必须关联用户和关卡
- XP 变更必须具备日志记录
- 解锁逻辑必须基于用户等级或通关状态

## 15. 开发规范

### 15.1 代码规范

- 前后端统一使用 `TypeScript`
- 统一启用 `ESLint` 与 `Prettier`
- 启用严格模式
- 禁止提交明显无用代码和未使用变量

### 15.2 命名规范

- 组件名使用 `PascalCase`
- 函数名使用 `camelCase`
- 常量名使用 `UPPER_SNAKE_CASE`
- 表名推荐复数形式，如 `users`、`levels`

### 15.3 接口规范

- 所有接口统一前缀为 `/api`
- 接口路径采用资源化设计
- 认证信息通过 `Authorization: Bearer <token>` 传递

### 15.4 Git 规范

- `main`：稳定分支
- `dev`：开发分支
- `feature/*`：功能分支
- `fix/*`：修复分支

推荐提交格式：

```text
feat: 新增世界地图页面
fix: 修复关卡提交判定错误
docs: 完善 spec 文档
refactor: 重构用户进度服务
```

## 16. 验收标准

### 16.1 产品验收

- 用户可注册并登录
- 世界地图可展示区域与解锁状态
- 至少 10 个关卡可进入
- 用户可在编辑器中编写 HTML/CSS/JS
- 预览区可展示运行结果
- 提交后系统可判定通关结果
- XP 与等级可正常增长
- 用户进度可正常保存

### 16.2 演示验收

- 可演示完整流程：登录 -> 选关 -> 编码 -> 运行 -> 提交 -> 通关 -> 升级
- 页面风格统一，具备游戏化视觉特征
- 核心流程无阻塞性错误

## 17. 风险与限制

- 关卡内容设计工作量较大
- 自动判题逻辑复杂度高
- 若首版直接实现后端真实运行环境，风险较高

当前控制策略：

- 首版聚焦前端闯关
- 后端先采用规则校验
- 后端和数据库高级区域先作为扩展内容预留

## 18. 迭代方向

- AI 教练系统
- 排行榜与成就系统
- React/Vue 框架专区
- Node/Spring Boot 后端关卡
- 数据库实战关卡
- Boss 战与综合项目挑战

## 19. 当前结论

当前阶段应以这份 `Spec` 为统一基线推进开发：

- 产品层面聚焦“地图 + 关卡 + 代码运行 + 成长反馈”
- 技术层面采用“前后端分离 + TypeScript + REST API”
- 实施层面优先打通 MVP 核心闭环，再逐步替换假数据与补齐高级能力
