# WebQuest 项目规范

## 1. 文档目的

本文档用于统一 `WebQuest` 项目的开发标准、协作规则、技术边界、目录规划与交付要求，确保产品设计、前端开发、后端开发、测试与答辩展示保持一致。

## 2. 项目概述

- 项目名称：`WebQuest`
- 项目定位：以 RPG 游戏化方式学习 Web 前端与后端开发的沉浸式在线学习平台
- 当前阶段：`V1.0 / MVP 设计阶段`
- 目标版本：先完成可演示、可操作、可通关的最小可行版本

## 3. 总体原则

### 3.1 产品原则

- 先完成核心学习闭环，再扩展高级玩法
- 先保证“可学习、可运行、可反馈”，再优化动画与视觉包装
- 所有功能必须服务于“闯关式学习 Web 开发”这一核心目标

### 3.2 开发原则

- 采用前后端分离架构
- 优先实现 MVP 功能，禁止在首版中堆叠高风险复杂功能
- 所有模块应具备清晰职责，避免耦合过高
- 所有接口、页面、数据表必须有明确命名规范
- 所有新增功能需同步补充文档说明

### 3.3 范围控制原则

MVP 阶段必须聚焦以下范围：

- 世界地图基础界面
- 10 个前端开发关卡
- HTML/CSS/JavaScript 在线编辑与实时预览
- 用户登录与学习进度记录
- XP 经验值与等级成长系统

MVP 阶段暂不实现：

- AI 教练系统
- 多人协作或排行榜
- 真实后端代码沙箱执行
- 多角色职业系统
- 高级复杂动画特效

## 4. 技术规范

### 4.1 技术选型

#### 前端

- 框架：`React`
- 构建工具：`Vite`
- 语言：`TypeScript`
- UI 方案：`Tailwind CSS` 或轻量组件库
- 代码编辑器：`Monaco Editor`
- 路由：`React Router`
- 状态管理：`Zustand` 或 `Redux Toolkit`

#### 后端

- 运行环境：`Node.js`
- 框架：`Express` 或 `NestJS`
- 接口风格：`REST API`
- 认证方式：`JWT`

#### 数据库

- 关系型数据库：`MySQL` 或 `PostgreSQL`
- 缓存：MVP 阶段可不接入 `Redis`

### 4.2 版本规范

- Node.js 版本建议：`18+`
- 包管理器统一为：`npm` 或 `pnpm`
- 团队必须统一包管理器，不允许混用锁文件

### 4.3 代码质量规范

- 前端与后端统一启用 `ESLint`
- 统一启用 `Prettier`
- TypeScript 开启严格模式
- 禁止提交明显无用代码、调试代码、未使用变量
- 公共工具函数需抽离，禁止页面中堆积重复逻辑

## 5. 项目目录规范

推荐项目结构如下：

```text
WebQuest/
  docs/
    项目规范.md
    需求文档.md
  web/
    src/
      assets/
      components/
      pages/
      layouts/
      router/
      store/
      hooks/
      services/
      utils/
      types/
  server/
    src/
      modules/
      controllers/
      services/
      middlewares/
      routes/
      models/
      utils/
      config/
  database/
    schema/
    seed/
```

目录说明：

- `docs`：项目文档
- `web`：前端项目
- `server`：后端项目
- `database`：数据库脚本、初始化数据与结构设计

## 6. 命名规范

### 6.1 通用命名

- 文件夹命名：小写短横线或语义化结构
- 组件文件命名：`PascalCase`
- 工具函数命名：`camelCase`
- 常量命名：`UPPER_SNAKE_CASE`
- 数据库表名：推荐使用复数形式，如 `users`、`levels`

### 6.2 页面命名建议

- `WorldMapPage`
- `LevelDetailPage`
- `BattleResultPage`
- `ProfilePage`
- `SkillTreePage`

### 6.3 接口命名规范

接口路径采用资源化设计，示例：

- `POST /api/auth/login`
- `GET /api/users/profile`
- `GET /api/zones`
- `GET /api/levels/:id`
- `POST /api/submissions`
- `GET /api/progress/current`

## 7. 前端开发规范

### 7.1 页面规范

- 每个页面必须职责清晰，避免一个页面承担过多业务逻辑
- 页面状态与业务状态分离
- 组件拆分优先按复用性与职责划分

### 7.2 UI 规范

- 统一使用游戏化视觉风格，但保持内容可读性
- 重要信息区分为：任务目标、编辑区域、运行预览、提交反馈
- 世界地图、关卡页、通关页必须形成统一视觉语言

### 7.3 编辑器与预览规范

- 编辑器区域必须支持 HTML、CSS、JavaScript 基础编辑
- 实时预览区域需与编辑区分栏展示
- 运行结果需可复现，不允许页面无提示地报错失败

## 8. 后端开发规范

### 8.1 接口规范

- 所有接口统一前缀：`/api`
- 返回数据采用统一结构

示例：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

- 错误返回也需结构统一
- 接口文档需同步维护

### 8.2 业务模块划分

- 用户模块
- 地图区域模块
- 关卡模块
- 提交记录模块
- XP 成长模块
- 技能树模块

### 8.3 安全规范

- 登录态必须进行鉴权校验
- 禁止信任前端直接传入的奖励结果
- XP 计算逻辑必须在后端完成
- 提交记录必须保留基础审计字段

## 9. 数据库设计规范

### 9.1 核心表建议

- `users`：用户信息
- `zones`：地图区域
- `levels`：关卡定义
- `submissions`：提交记录
- `user_progress`：用户关卡进度
- `xp_logs`：经验值记录
- `skills`：技能树节点
- `user_skills`：用户技能解锁记录

### 9.2 字段规范

- 每张表必须有主键 `id`
- 必须包含 `created_at`、`updated_at`
- 状态字段统一命名为 `status`
- 布尔值字段采用 `is_` 前缀，如 `is_unlocked`

### 9.3 数据约束

- 关卡必须归属某个地图区域
- 用户 XP 变更必须有日志记录
- 提交记录必须关联用户和关卡

## 10. 测试规范

### 10.1 测试范围

- 前端关键页面渲染测试
- 关卡提交流程测试
- 登录与鉴权测试
- XP 增长逻辑测试
- 关卡解锁逻辑测试

### 10.2 验证重点

- 编辑器是否可正常输入
- 预览区是否正常执行
- 提交后是否能正确判断通关
- 通关后是否正确发放 XP
- 下一关是否按规则解锁

## 11. Git 协作规范

### 11.1 分支建议

- `main`：稳定主分支
- `dev`：开发分支
- `feature/*`：功能分支
- `fix/*`：问题修复分支

### 11.2 提交规范

推荐提交格式：

```text
feat: 新增世界地图页面
fix: 修复关卡提交判定错误
docs: 完善需求文档
refactor: 重构用户进度服务
```

## 12. 文档规范

- 需求变更必须同步更新需求文档
- 数据结构调整必须同步更新数据库设计说明
- 核心流程变化必须补充流程图或说明
- 所有答辩展示内容必须与当前项目实现一致

## 13. 交付规范

MVP 阶段必须完成以下交付物：

- 产品需求文档
- 项目规范文档
- 页面原型或界面设计图
- 数据库结构说明
- 前后端代码仓库
- 项目演示说明

## 14. 附录

### 14.1 项目目标总结

`WebQuest` 的目标不是做一个普通教程网站，而是搭建一个具有“地图探索、关卡挑战、代码实战、成长反馈”闭环的游戏化 Web 学习平台。

### 14.2 当前实施建议

首版优先完成：

1. 世界地图基础界面
2. 关卡页编辑器与预览区
3. 提交判定与通关反馈
4. XP 与等级系统
5. 用户进度存储
